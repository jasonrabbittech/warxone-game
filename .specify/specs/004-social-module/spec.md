# Feature Specification: Social Module / 社交模块

**Feature Branch**: `004-social-module`

**Created**: 2026-06-27

**Status**: Draft

**Input**: User description: "定义Social 模块"

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 游戏内聊天系统 (Priority: P1)

玩家需要与其他玩家进行实时交流，包括全球聊天、房间聊天和私信功能。聊天系统帮助玩家建立社区感、协调游戏策略、并增强游戏的整体社交体验。

**Why this priority**: 聊天是游戏社交功能的基础，为其他社交功能（好友、联盟）提供沟通渠道。没有聊天系统，玩家之间的互动将非常有限。

**Independent Test**: 可以独立测试聊天功能 - 部署聊天系统后，玩家可以发送和接收消息，验证消息传递、显示和存储功能。

**Acceptance Scenarios**:

1. **Given** 玩家已登录游戏, **When** 玩家在聊天界面输入消息并发送, **Then** 消息应显示在全局聊天频道中，所有在线玩家都能看到
2. **Given** 玩家在多人对战房间中, **When** 玩家发送消息, **Then** 消息只显示在该房间内的玩家
3. **Given** 玩家查看聊天记录, **When** 滚动聊天历史, **Then** 应显示最近的 100 条消息
4. **Given** 玩家尝试发送不当内容, **When** 消息包含禁用词汇, **Then** 消息应被屏蔽或替换，并提示玩家
5. **Given** 玩家离线, **When** 其他玩家发送私信, **Then** 私信应被保存，玩家下次登录时能看到

---

### User Story 2 - 好友系统 (Priority: P2)

玩家需要添加、管理和与好友互动。好友系统允许玩家建立个人社交网络，邀请好友对战，并跟踪好友的游戏活动。

**Why this priority**: 好友系统是建立长期玩家关系的关键。它鼓励玩家返回游戏与好友一起玩，提高玩家留存率。

**Independent Test**: 可以独立测试好友功能 - 玩家可以发送好友请求、接受/拒绝请求、查看好友列表，所有这些功能不依赖其他社交功能。

**Acceptance Scenarios**:

1. **Given** 玩家在搜索界面, **When** 输入其他玩家的用户名并发送好友请求, **Then** 目标玩家应收到好友请求通知
2. **Given** 玩家收到好友请求, **When** 点击接受, **Then** 两位玩家应互相出现在对方的好友列表中
3. **Given** 玩家查看好友列表, **When** 查看在线状态, **Then** 应显示每位好友的在线/离线状态和最后上线时间
4. **Given** 玩家与好友在对战房间中, **When** 点击好友头像, **Then** 应显示好友的详细信息（等级、胜率、资源等）
5. **Given** 玩家想与好友对战, **When** 从好友列表发送对战邀请, **Then** 好友应收到邀请通知，可以接受或拒绝

---

### User Story 3 - 联盟系统 (Priority: P3)

玩家可以创建或加入联盟，与联盟成员协作，参与联盟战争，并在联盟排行榜上竞争。联盟系统为大批量玩家提供归属感和团队玩法。

**Why this priority**: 联盟系统是高级社交功能，适合长期玩家。它在好友系统基础上提供更复杂的社交互动，但需要好友系统作为基础。

**Independent Test**: 可以独立测试联盟功能（假设好友系统已实现）- 玩家可以创建联盟、邀请成员、参与联盟聊天、查看联盟排行榜。

**Acceptance Scenarios**:

1. **Given** 玩家想创建联盟, **When** 填写联盟名称、描述和标签并提交, **Then** 新联盟应被创建，玩家自动成为联盟首领
2. **Given** 玩家查看联盟列表, **When** 搜索或浏览联盟, **Then** 应显示联盟名称、成员数量、活跃度和描述
3. **Given** 联盟成员在联盟聊天中, **When** 发送消息, **Then** 所有联盟成员都能看到消息
4. **Given** 联盟首领管理联盟, **When** 任命官员、踢出成员或修改联盟设置, **Then** 更改应立即生效并通知相关成员
5. **Given** 联盟参与联盟战争, **When** 战争期间与敌对联盟对战, **Then** 应记录战争积分，战争结束后宣布获胜联盟

---

### Edge Cases

- 聊天系统：当聊天消息量非常大时（如 1000+ 条/分钟），系统如何处理？是否限制消息频率？
- 好友系统：当玩家发送好友请求但对方已达到好友上限（如 100 人）时，如何处理？
- 联盟系统：当联盟首领长期不在线（如 30 天），是否自动转移首领权限？
- 私信系统：当玩家收到大量私信（如垃圾消息），如何防止骚扰？
- 聊天审查：如何平衡内容审查和玩家自由表达？是否有申诉机制？
- 跨服务器：如果游戏有多个服务器，聊天和好友是否跨服务器？

## Requirements *(mandatory)*

### Functional Requirements

#### 聊天功能 (Chat)

- **FR-001**: 系统必须提供全局聊天频道，所有在线玩家可以发送和接收消息
- **FR-002**: 系统必须提供房间聊天频道，仅对战房间内的玩家可以交流
- **FR-003**: 系统必须提供私信功能，玩家可以一对一发送私人消息
- **FR-004**: 系统必须支持聊天消息持久化，保存最近 100 条消息供新加入玩家查看
- **FR-005**: 系统必须提供消息频率限制，防止垃圾消息（如每分钟最多 10 条消息）
- **FR-006**: 系统必须提供不当内容过滤，自动屏蔽或替换禁用词汇
- **FR-007**: 系统必须支持聊天表情符号，玩家可以在消息中插入表情
- **FR-008**: 系统必须显示玩家在线状态和最后上线时间

#### 好友功能 (Friends)

- **FR-009**: 系统必须允许玩家通过用户名或玩家 ID 搜索其他玩家
- **FR-010**: 系统必须允许玩家发送、接受、拒绝和删除好友请求
- **FR-011**: 系统必须维护好友列表，显示每位好友的在线状态和基本信息
- **FR-012**: 系统必须允许玩家从好友列表直接发送对战邀请
- **FR-013**: 系统必须限制好友数量上限（默认 100 人），防止列表过大影响性能
- **FR-014**: 系统必须记录好友互动历史，如最后对战时间、最后聊天时间

#### 联盟功能 (Alliances)

- **FR-015**: 系统必须允许玩家创建联盟，设置名称、描述、标签和加入要求
- **FR-016**: 系统必须允许玩家搜索、浏览和申请加入现有联盟
- **FR-017**: 系统必须提供联盟角色系统，包括首领、官员和普通成员
- **FR-018**: 系统必须提供联盟聊天频道，仅联盟成员可以参与
- **FR-019**: 系统必须维护联盟成员列表，记录每位成员的加入时间和贡献度
- **FR-020**: 系统必须提供联盟排行榜，根据联盟总积分、成员数量或战争胜率排名
- **FR-021**: 系统必须支持联盟战争功能，联盟之间可以发起或接受战争挑战
- **FR-022**: 系统必须限制联盟成员数量上限（默认 50 人），防止过大影响管理

### Key Entities

- **ChatMessage**: 表示一条聊天消息，包含发送者 ID、内容、时间戳、频道类型（全局/房间/私信）和接收者 ID（如果是私信）
- **Friendship**: 表示两个玩家之间的好友关系，包含发起者 ID、接受者 ID、建立时间和互动记录
- **Alliance**: 表示联盟，包含名称、描述、标签、创建者 ID、成员列表、创建时间和联盟设置
- **AllianceMember**: 表示联盟成员，包含玩家 ID、联盟 ID、角色、加入时间和贡献度
- **AllianceWar**: 表示联盟战争，包含进攻联盟 ID、防守联盟 ID、开始时间、结束时间和战争状态

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 玩家可以发送聊天消息并在 2 秒内看到自己的消息显示（消息传递延迟低于 2 秒）
- **SC-002**: 聊天系统支持至少 500 名玩家同时在线聊天，消息传递成功率高于 99%
- **SC-003**: 玩家可以在 30 秒内完成添加好友的完整流程（搜索、发送请求、接受）
- **SC-004**: 好友列表加载时间在 1 秒以内，即使玩家有最大允许的好友数量（100 人）
- **SC-005**: 联盟创建流程在 1 分钟内完成，包括填写信息和确认创建
- **SC-006**: 80% 的新玩家在注册后 7 天内添加至少 1 名好友（好友转化率）
- **SC-007**: 玩家日均发送聊天消息数达到 5 条以上（聊天活跃度）
- **SC-008**: 联盟成员日均登录率高于非联盟玩家 20%（联盟留存效果）
- **SC-009**: 系统可以自动过滤 95% 以上的不当内容（内容审查有效率）
- **SC-010**: 玩家对社交功能的满意度评分达到 4.0/5.0 以上（用户满意度）

## Assumptions

- 假设玩家有稳定的互联网连接，可以支持实时聊天功能
- 假设游戏已有用户认证系统，社交功能可以复用现有用户 ID 和认证机制
- 假设实时通信使用托管的云服务，不自行搭建服务器
- 假设聊天消息存储使用缓存层和持久化存储的组合方式，以平衡性能和历史记录查询
- 假设联盟功能在好友系统之后实现，因为联盟邀请依赖好友关系
- 假设内容审查使用自动关键词过滤，暂不引入 AI 内容审查（降低复杂度和成本）
- 假设移动端和桌面端都需要支持社交功能，界面需要响应式设计
- 假设多语言支持（i18n）适用于聊天界面、好友请求通知和联盟描述
- 假设联盟战争功能在 v1 版本中可能是简化版（如仅记录积分，无实时对战）
- 假设玩家数据隐私需要遵守相关法律法规，私信内容需要加密存储
